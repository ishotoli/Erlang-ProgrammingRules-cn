Program Development Using Erlang -
编程规则与约定

摘要

这是一篇使用Elang开发时的编程规范和建议

注：这只是个初步文档而且还未完成

The requirements for the use of EBC's "Base System" are not documented here, but must be followed at a very early design phase if the "Base System" is to be used. These requirements are documented in 1/10268-AND 10406 Uen "MAP - Start and Error Recovery".

1 目的

本文列举了一些在使用Erlang语言进行设计和编码过程中需要考虑的一些因素，这些规范只适用于Erlang语言，并不是一种通用规范。

2 结构和Erlang术语

Erlang 系统一般是用modules为单位组成的。 模块又是由函数 与 属性组成。函数默认是模块内可见的，也可以被导出，也就是说它们导出后可以被其它模块调用。属性以 "-" 开始而且一般放在模块开头的位置。

使用Erlang语言设计的系统都是以 进程 为单位 工作 的. 一个进程就是一个使用多个模块和函数组成的任务。进程之间的通讯靠 发送 消息。 进程 接收 发给它们的消息，每个进程都可以决定当前准备接收哪个消息。其他消息会排队直到进程准备接收处理它们。

一个进程可以通过建立一个链接到它监督另一个进程的存在。当进程结束的时候，他会自动给连接的进程发送一个退出信号。进程接收到退出信号的默认处理是终止进程并将退出信号传递给其他链接的进程。进程也可以通过捕获退出改变这个默认行为，这将把所有的退出信号变为普通的消息来处理。

纯函数 是一种无论上下文环境，给出的同样的参数都能返回相同值的一种函数。在函数式编程中这通常是我们希望看到的。那么非纯函数就意味着可能有副作用。

有以下行为的函数有可能产生副作用 a) 发送消息 b) 接收消息 c) 调用 退出 事件 d) 调用了任何可能改变进程环境或模式的内置函数（例如get/1, put/2, erase/1, process_flag/2 等等) e)使用了ETS(add by kaiz)。

3 SW工程原理

3.1 从一个模块（module）中尽可能少的导出方法（functions）

Modules are the basic code structuring entity in Erlang. A module can contain a large number of functions but only functions which are included in the export list of the module can be called from outside the module. 模块是Erlang中基本的代码结构单位。一个模块包含很多函数，但只有在模块导出列表中的函数才能在外部被调用。

Seen from the outside the complexity of a module depends upon the number of functions which are exported from the module. A module which exports one or two functions is usually easier to understand than a module which exports dozens of functions. 从外面看到的一个模块的复杂性取决于从模块导出的函数的数量。一个只导出了一两个函数的模块通常比导出了几十个函数的模块要简单。

Modules where the ratio of exported/non-exported functions is low are desirable in that a user of the module only needs to understand the functionality of the functions which are exported from the module. 通常情况下，低导出/非导出函数比例的模块更受欢迎，因为模块的用户只需要了解模块导出函数的功能就够了。

In addition, the writer or maintainer of the code in the module can change the internal structure of the module in any appropriate manner provided the external interface remains unchanged. 此外，该模块中的作者或代码的维护者可以在不修改已导出函数的情况下修改模块的内部结构

3.2 尽量减少模块间的依赖关系

A module which calls functions in many different modules will be more difficult to maintain than a module which only calls functions in a few different modules. 一个模块如果调用很多不同模块会比调用很少其他模块难维护。

This is because each time we make a change to a module interface, we have to check all places in the code where this module is called. Reducing the interdependencies between modules simplifies the problem of maintaining these modules. 这是因为每次我们修改一个模块接口时，我们不得不检查所有调用这个模块的代码。减少模块之间的依赖性会简化模块的维护问题。

We can simplify the system structure by reducing the number of different modules which are called from a given module. 我们可以通过减少从一个指定的模块调用其它不同模块的数量简化系统结构

Note also that it is desirable that the inter-module calling dependencies form a tree and not a cyclic graph. Example: 同样需要注意的是，理想状态下模块间调用依赖性应该形成树形，而不是一个循环图

But not 不要这样

3.3 把常用的代码放到库

Commonly used code should be placed into libraries. The libraries should be collections of related functions. Great effort should be made in ensuring that libraries contain functions of the same type. Thus a library such as lists containing only functions for manipulating lists is a good choice, whereas a library, lists_and_maths containing a combination of functions for manipulating lists and for mathematics is a very bad choice. 常用的代码应放入库中。这些库应该是相关功能集合。应该努力确保库包含相同类型的函数。因此，一个lists库只包含操作列表的函数是一个不错的选择，而 lists_and_maths 库同时包含列表操作和数学运算则是一个糟糕的选择。

The best library functions have no side effects. Libraries with functions with side effects limit the re-usability. 库函数最好不要有任何的副作用，有副作用的库和函数会大大限制自身的重用度。

3.4 隔离"刁钻" 或 "脏"的代码到独立的模块里

Often a problem can be solved by using a mixture of clean and dirty code. Separate the clean and dirty code into separate modules. 通常一些问题可以通过混合使用清洁的和脏的代码来解决。最好将清洁的和脏的代码分离到不同的模块当中。

Dirty code is code that does dirty things. Example: 脏代码就是用来处理脏事务的，例如：

  • Uses the process dictionary. 使用了进程字典。（应当包括ETS）

  • Uses erlang:process_info/1 for strange purposes. 为了特殊目的使用了erlang:process_info/1

  • Does anything that you are not supposed to do (but have to do). 不得不做一些你预期之外的事情

Concentrate on trying to maximize the amount of clean code and minimize the amount of dirty code. Isolate the dirty code and clearly comment or otherwise document all side effects and problems associated with this part of the code. 集中于将清洁代码量最大化和脏的代码量最小化，隔离脏的代码，并明确注释或以其他方式记录在这部分代码相关联的所有副作用和问题。

3.5 不要假设调用者会如何处理方法返回的结果

Don't make assumptions about why a function has been called or about what the caller of a function wishes to do with the results. 不要去假设为什么会调用这个函数或者这个函数的调用者会拿结果去做什么。

For example, suppose we call a routine with certain arguments which may be invalid. The implementer of the routine should not make any assumptions about what the caller of the function wishes to happen when the arguments are invalid. 例如，假设我们调用一个子程序，所带的参数可能是无效的，这段子程序的编写者不应该假设调用者希望参数无效的时候希望发生什么。

Thus we should not write code like 因此，我们不应该这样写代码： 


do_something(Args) -> 
  case check_args(Args) of 
    ok -> 
      {ok, do_it(Args)}; 
    {error, What} -> 
      String = format_the_error(What), 
      io:format("* error:~s\n", [String]), %% 别这么做
      error 
  end.

Instead write something like: 而应该这样： 

do_something(Args) ->
  case check_args(Args) of
    ok ->
      {ok, do_it(Args)};
    {error, What} ->
      {error, What}
  end.

error_report({error, What}) ->
  format_the_error(What).

In the former case the error string is always printed on standard output, in the latter case an error descriptor is returned to the application. The application can now decide what to do with this error descriptor. 在前一种情况下错误字符串始终打印在标准输出，在后一种情况下的错误描述符返回到应用。应用这时可以决定怎样去处理错误描述符。

By calling error_report/1 the application can convert the error descriptor to a printable string and print it if so required. But this may not be the desired behavior - in any case the decision as to what to do with the result is left to the caller. 通过调用error_report/1应用程序可以将错误描述符转化为可打印字符，如果需要的话可以将它打印出来。但这也许不是期望的行为，在任何情况下怎样处理这些结果的决定权应该留给调用者。

3.6 从代码或行为中抽象出通用的模式

Whenever you have the same pattern of code in two or more places in the code try to isolate this in a common function and call this function instead of having the code in two different places. Copied code requires much effort to maintain. 只要你在两个以上的地方使用相同模式的代码，将它们封装成一个通用函数，用调用函数的方式取代同样的代码在两个不同的地方使用。通常拷贝代码更难维护。

If you see similar patterns of code (i.e. almost identical) in two or more places in the code it is worth taking some time to see if one cannot change the problem slightly to make the different cases the same and then write a small amount of additional code to describe the differences between the two. 如果你在两个以上的地方看到相似模式的代码（例如几乎一样），值得花一些时间，看看是否不改变问题的情况下使稍微有不同的代码变得相同，然后编写额外的少量代码来描述两者之间的差异。

Avoid "copy" and "paste" programming, use functions! 避免"拷贝"和"粘贴"的编码方式，用函数！

3.7 自顶向下

Write your program using the top-down fashion, not bottom-up (starting with details). Top-down is a nice way of successively approaching details of the implementation, ending up with defining primitive functions. The code will be independent of representation since the representation is not known when the higher levels of code are designed. 用自顶向下的方式编程，不要用自下向上的方式（从细节开始）。自顶向下是一种逐步达成实现细节，以定义基本函数结束的非常好的方式。代码是独立表现的，直到设计出不考虑代码表现的更高层代码。

3.8 不要优化代码

Don't optimize your code at the first stage. First make it right, then (if necessary) make it fast (while keeping it right). 不要从一开始就优化你的代码，首先让它正确运行，然后（必要的话）让它更快（必须保持正确）。

3.9 使用“最少惊动”原则

The system should always respond in a manner which causes the "least astonishment" to the user - i.e. a user should be able to predict what will happen when they do something and not be astonished by the result. 系统总应该以"最少惊动"用户的原则响应，比方说用户应该能够预测他们做一些事情会发生什么，而不是被结果惊动到。

This has to do with consistency, a consistent system where different modules do things in a similar manner will be much easier to understand than a system where each module does things in a different manner. 这个具有一致性，一致性的系统中不同模块使用相同的原则比系统每个模块使用不同的原则要更容易理解。

If you get astonished by what a function does, either your function solves the wrong problem or it has a wrong name. 如果你被函数的行为惊动到了，一般不是你的函数功能有问题，就是函数命名有问题。

3.10 设法消除副作用

Erlang has several primitives which have side effects. Functions which use these cannot be easily re-used since they cause permanent changes to their environment and you have to know the exact state of the process before calling such routines. Erlang有一些原函数有副作用。使用了这些原函数的函数不能轻易被重用，因为他们会导致环境永久性的变化，你在调用这些子过程之前需要了解这些进程的确切状态。

Write as much as possible of the code with side-effect free code. 尽量编写没有副作用的代码。

Maximize the number of pure functions. 将纯函数的数量最大化。

Collect together the functions which have side effect and clearly document all the side effects. 将具有副作用的函数集中在一起，清晰的记录所有的副作用。

With a little care most code can be written in a side-effect free manner - this will make the system a lot easier to maintain, test and understand. 随着一点点的关心大多数代码可以以无副作用的方式被写入 - 这将使系统更容易维护，测试和理解。

3.11 不要让私有数据结构“泄漏”出模块

This is best illustrated by a simple example. We define a simple module called queue - to implement queues: 这最好通过一个简单的例子展示。我们定义了一个简单的模块叫做 queue 来实现队列： 

-module(queue).
-export([add/2, fetch/1]).

add(Item, Q) -> 
  lists:append(Q, [Item]).

fetch([H|T]) -> 
  {ok, H, T}; 
fetch([]) -> 
  empty.

This implements a queue as a list, unfortunately to use this the user must know that the queue is represented as a list. A typical program to use this might contain the following code fragment: 这个模块用列表实现了一个队列，不幸的是用户必须知道这个队列是用列表实现的。使用这个模块的程序通常会包含如下代码片段： 

NewQ = [], % 千万别这样
Queue1 = queue:add(joe, NewQ), 
Queue2 = queue:add(mike, Queue1), ....

This is bad - since the user a) needs to know that the queue is represented as a list and b) the implementer cannot change the internal representation of the queue (this they might want to do later to provide a better version of the module). 这很糟糕，a）用户需要知道这个队列是以列表的方式实现的；b）开发者不能更换queue模块的内部实现（他们很可能希望稍晚一些提供一个更好的版本）

Better is: 最好是这样： 

-module(queue).
-export([new/0, add/2, fetch/1]).

new() -> 
  [].

add(Item, Q) -> 
  lists:append(Q, [Item]).

fetch([H|T]) -> 
  {ok, H, T}; 
fetch([]) -> 
  empty.

Now we can write: 现在我们可以这样使用： 

NewQ = queue:new(), 
Queue1 = queue:add(joe, NewQ), 
Queue2 = queue:add(mike, Queue1), ...

Which is much better and corrects this problem. Now suppose the user needs to know the length of the queue, they might be tempted to write: 这样就可以很好的修正了这个问题。假设用户需要知道队列长度，他们或许会这样去写： 

Len = length(Queue) % 千万别这样

since they know that the queue is represented as a list. Again this is bad programming practice and leads to code which is very difficult to maintain and understand. If they need to know the length of the queue then a length function must be added to the module, thus: 只要他们知道队列是用list实现的。再次说明这是一种很糟糕的编程实践，会导致代码很难理解和维护。如果他们需要知道队列的长度，就需要往模块中增加一个获取长度的函数，例如： 

-module(queue).
-export([new/0, add/2, fetch/1, len/1]).

new() -> [].

add(Item, Q) ->
  lists:append(Q, [Item]).

fetch([H|T]) -> 
  {ok, H, T}; 

fetch([]) -> 
  empty.

len(Q) -> 
  length(Q).

Now the user can call queue:len(Queue) instead. 现在我们可以直接是用queue:len(Queue)来获取长度。

Here we say that we have "abstracted out" all the details of the queue (the queue is in fact what is called an "abstract data type"). 险遭我们可以说我们"抽象"出来了queue的所有细节（实际上队列被称为"抽象数据类型"）。

Why do we go to all this trouble? - the practice of abstracting out internal details of the implementation allows us to change the implementation without changing the code of the modules which call the functions in the module we have changed. So, for example, a better implementation of the queue is as follows: 为什么我们要自找麻烦？抽象出了实施内部细节的做法使我们能够改变实现而不改变调用其中的函数的代码。举个例子，队列一种更好的实现方式是这样子的： 

-module(queue).
-export([new/0, add/2, fetch/1, len/1]).

new() -> 
  {[],[]}.

add(Item, {X,Y}) -> % 快速添加对象
  {[Item|X], Y}.

fetch({X, [H|T]}) -> 
  {ok, H, {X,T}}; 

fetch({[], []) -> 
  empty; 

fetch({X, []) -> 
  % 有时会加重计算量。
  fetch({[],lists:reverse(X)}).

len({X,Y}) -> 
  length(X) + length(Y).

3.12 尽量使代码确定

A deterministic program is one which will always run in the same manner no matter how many times the program is run. A non-deterministic program may deliver different results each time it is run. For debugging purposes it is a good idea to make things as deterministic as possible. This helps make errors reproducible. 一个确定性的程序不管运行多少次，都将是以相同的方式运行。一个非确定性的程序每次运行都可能产生不同的结果。从调试角度来说应该尽可能的让事情变成确定性的。这让错误变得可重现。

For example, suppose one process has to start five parallel processes and then check that they have started correctly, suppose further that the order in which these five are started does not matter. 举个例子，假设一个进程启动了五个并行进程然后检查它们是否正确启动，进一步假设这五个进程的启动顺序并无影响。

We could then choose to either start all five in parallel and then check that they have all started correctly but it would be better to start them one at a time and check that each one has started correctly before starting the next one. 然后，我们可以选择要么并行启动所有五个，然后检查他们是否都正确启动，但更好的做法是启动一个之后检查然后再启动下一个。

3.13 不要 "防御性编程"

A defensive program is one where the programmer does not "trust" the input data to the part of the system they are programming. In general one should not test input data to functions for correctness. Most of the code in the system should be written with the assumption that the input data to the function in question is correct. Only a small part of the code should actually perform any checking of the data. This is usually done when data "enters" the system for the first time, once data has been checked as it enters the system it should thereafter be assumed correct. 防御性变成是一种编程方式，开发者"不信任"系统给出的输入数据。一般来说不应该检查函数输入数据的正确性。系统中的大部分代码应该假设输入数据是正确的。只有少数部分代码需要对数据进行检查。这个通常是在数据第一次"输入"系统时进行，一旦数据通过了检查进入系统就应该认为它是正确的。

Example:例如 

%% Args: Option is all|normal
get_server_usage_info(Option, AsciiPid) ->
  Pid = list_to_pid(AsciiPid),
  case Option of
    all -> get_all_info(Pid);
    normal -> get_normal_info(Pid)
  end.

The function will crash if Option neither normal nor all, and it should do that. The caller is responsible for supplying correct input. 当参数Option不为normal或者all的时候这个函数会崩溃，应该要这样。调用者有责任提供正确的函数。

3.14 将硬件接口封装成设备驱动

Hardware should be isolated from the system through the use of device drivers. The device drivers should implement hardware interfaces which make the hardware appear as if they were Erlang processes. Hardware should be made to look and behave like normal Erlang processes. Hardware should appear to receive and send normal Erlang messages and should respond in the conventional manner when errors occur. 硬件应该从系统通过使用设备驱动程序来分离。设备驱动程序应该实现硬件接口，它使硬件看起来好像也是Erlang进程。硬件应使外观和行为像正常Erlang进程。硬件应该正常的接收和发送消息，并在发生错误时以传统的方式响应。

3.15 同一个函数中的“要”和“不要”

Suppose we have a program which opens a file, does something with it and closes it later. This should be coded as: 假设我们有一个程序需要打开文件，处理一些事情然后关掉它。通常来说应该是这样编码： 

do_something_with(File) -> 
  case file:open(File, read) of, 
    {ok, Stream} ->
      doit(Stream), 
      file:close(Stream) % 正常处理
    Error -> Error
  end.

Note the symmetry in opening the file (file:open)and closing it (file:close) in the same routine. The solution below is much harder to follow and it is not obvious which file that is closed. Don't program it like this: 注意打开文件的(file:open)和关闭文件的(file:close)在同一段代码中是对称的。下面的解决方案更难理解，而且文件是否被关闭很不明显，不要这样编程： 

do_something_with(File) -> 
  case file:open(File, read) of, 
    {ok, Stream} ->
      doit(Stream)
    Error -> Error
  end.

doit(Stream) -> 
  ...., 
  func234(...,Stream,...).
  ...

func234(..., Stream, ...) ->
  ...,
  file:close(Stream) %% 千万别这样

4 错误处理

4.1 分离错误处理和正常逻辑处理

Don't clutter code for the "normal case" with code designed to handle exceptions. As far as possible you should only program the normal case. If the code for the normal case fails, your process should report the error and crash as soon as possible. Don't try to fix up the error and continue. The error should be handled in a different process (See "Each process should only have one "role"" on page 15.). 不要将正常代码和异常处理代码混到一起。你应该尽可能的只处理正常情况，如果正常情况处理出错了，你的进程应该尽快汇报错误然后崩溃掉。不要尝试修正错误然后继续。错误应该在另一个进程中处理(参见"每个进程只扮演一个"角色"")

Clean separation of error recovery code and normal case code should greatly simplify the overall system design. 错误恢复代码和正确情况代码清晰的分离后，整个系统设计应该会大大简化。

The error logs which are generated when a software or hardware error is detected will be used at a later stage to diagnose and correct the error. A permanent record of any information that will be helpful in this process should be kept. 当检测到的软件或硬件错误将产生错误日志，在稍后阶段使用来诊断和纠正错误。有助于这个过程中的任何信息都应保存永久记录。

4.2 识别错误根源

One of the basic elements of system design is identifying which part of the system has to be correct and which part of the system does not have to be correct. 在系统设计中的一个基本元素是识别系统的那一部分是正常的，那一部分是有问题的。

In conventional operating system design the kernel of the system is assumed to be, and must be, correct, whereas all user application programs do not necessarily have to be correct. If a user application program fails this will only concern the application where the failure occurred but should not affect the integrity of the system as a whole. 在常规的操作系统中，内核系统设计时被假定为，而且必须是正确的，而所有用户应用程序不一定是正确的。如果用户应用程序失败，这将仅涉及其中发生的故障，但不应影响系统作为一个整体的完整应用。

The first part of the system design must be to identify that part of the system which must be correct; we call this the error kernel. Often the error kernel has some kind of real-time memory resident data base which stores the state of the hardware. 系统设计的第一部分必须识别系统那一部分是正常的；我们称之为错误内核。通常错误内核有某种实时内存持久化数据库用来存储硬件状态。

5 进程, 服务和消息

5.1 在一个模块中实现一个进程

Code for implementing a single process should be contained in one module. A process can call functions in any library routines but the code for the "top loop" of the process should be contained in a single module. The code for the top loop of a process should not be split into several modules - this would make the flow of control extremely difficult to understand. This does not mean that one should not make use of generic server libraries, these are for helping structuring the control flow. 实现一个独立进程的代码应该都包含在一个模块中。进程可以调用多个库中的函数，但进程"顶循环"部分的代码应该包含在一个单独模块中。进程定循环的代码不应被分割到多个多个模块中，这通常会导致流程控制很难理解。这并不意味着需要做一个通用服务库，这样处理可以帮助架构控制流。

Conversely, code for no more than one kind of process should be implemented in a single module. Modules containing code for several different processes can be extremely difficult to understand. The code for each individual process should be broken out into a separate module. 反过来，只被一种进程调用的代码需要实现在单一模块中。模块包含为多种进程调用的代码会很难理解。每个独立进程的代码应该分割到不同的模块中。

5.2 使用进程来构建系统

Processes are the basic system structuring elements. But don't use processes and message passing when a function call can be used instead. 进程是基本系统结构元素。但在函数调用可以替代的时候，不要使用进程和消息。

5.3 注册进程

Registered processes should be registered with the same name as the module. This makes it easy to find the code for a process. 注册进程在注册时应当使用和模块相同的名称，这会让找到进程对应的代码变得更容易。

Only register processes that should live a long time. 只有会生存很长时间的进程才需要注册。

5.4 为系统中每一个活动分配唯一的进程

When deciding whether to implement things using sequential or parallel processes then the structure implied by the intrinsic structure of the problem should be used. The main rule is: 当你决定使用顺序或并行进程实现的时候，应该由该问题固有结构所隐含的结构来实现。主要的规则是：

"Use one parallel process to model each truly concurrent activity in the real world" "用一个并行进程来模拟真实世界中每一个真正的并发活动 "

If there is a one-to-one mapping between the number of parallel processes and the number of truly parallel activities in the real world, the program will be easy to understand. 如果在真实世界中的实际并发活动数和并发进程数是一一对应的，程序代码就非常易于理解。

5.5 每个进程只扮演一个 "角色"

Processes can have different roles in the system, for example in the client-server model. 进程可以在系统中扮演不同的角色，例如客户端-服务器模型。

As far as possible a process should only have one role, i.e. it can be a client or a server but should not combine these roles. 一个进程尽可能的只有一个角色，例如它可以是客户端或者服务器，但不要把两者混合。

Other roles which process might have are: 进程其它的角色可能是：

Supervisor: watches other processes and restarts them if they fail. 监控进程：监控其它进程，如果其他进程产生错误了就重启它们。
Worker: a normal work process (can have errors). 工作进程：一个普通的工作进程（可以出错）。
Trusted Worker: not allowed to have errors. 可信工作进程：不允许有任何错误。

5.6 只要有可能就要在服务和协议解析中使用通用函数

In many circumstances it is a good idea to use generic server programs such as the generic server implemented in the standard libraries. Consistent use of a small set of generic servers will greatly simplify the total system structure. 在许多情况下，使用通用的服务程序例如标准库当中的通用服务，都是一个好主意。一致使用一套通用的服务，将大大简化整个系统的结构。

The same is possible for most of the protocol handling software in the system. 在系统中大部分协议处理软件也是一样。

5.7 给消息打标签

All messages should be tagged. This makes the order in the receive statement less important and the implementation of new messages easier. 所有的消息都要有标签。这会降低消息接收处理时的顺序不那么重要，添加新消息也可以更简单。

Don't program like this: 

loop(State) ->
  receive
    ...
    {Mod, Funcs, Args} -> % 千万别这样
      apply(Mod, Funcs, Args},
      loop(State);
    ...
  end.

The new message {get_status_info, From, Option} will introduce a conflict if it is placed below the {Mod, Func, Args} message. 新消息{get_status_info, From, Option}如果放在{Mod, Func, Args}消息处理之后将会产生冲突。

If messages are synchronous, the return message should be tagged with a new atom, describing the returned message. Example: if the incoming message is tagged get_status_info, the returned message could be tagged status_info. One reason for choosing different tags is to make debugging easier. 如果消息是同步的，那么返回的消息应该使用一个新原子打上标签，描述这是一个返回的消息。例如：如果接受的消息标签是get_status_info，那么返回的消息标签可以使status_info。使用不同标签的一个原因是使调试更容易一些。

This is a good solution: 这是个好方案： 

loop(State) ->
  receive
    ...
    {execute, Mod, Funcs, Args} -> % 使用带标签的消息
      apply(Mod, Funcs, Args},
      loop(State);
    {get_status_info, From, Option} ->
      From ! {status_info, get_status_info(Option, State)},
      loop(State);    
    ...
  end.

5.8 处理未知消息

Every server should have an Other alternative in at least one receive statement. This is to avoid filling up message queues. Example: 每一个服务应该在receive代码段中包含一个其它额外处理。这样可以避免消息队列被填满。例如： 

main_loop() ->
  receive
    {msg1, Msg1} -> 
      ...,
      main_loop();
    {msg2, Msg2} ->
      ...,
      main_loop();
    Other -> % 清理消息队列
      error_logger:error_msg(
          "Error: Process ~w got unknown msg ~w~n.", 
          [self(), Other]),
      main_loop()
  end.

5.9 编写尾递归服务器

All servers must be tail-recursive, otherwise the server will consume memory until the system runs out of it. 所有的服务必须是尾递归的，否则服务会一直消耗内存直到系统用尽。

Don't program like this: 不要像这样编程： 

loop() ->
  receive
    {msg1, Msg1} -> 
      ...,
      loop();
    stop ->
      true;
    Other ->
      error_logger:log({error, {process_got_other, self(), Other}}),
      loop()
  end,
  io:format("Server going down").                % 千万不要这样 
                % 这不是尾递归

This is a correct solution: 这是个正确的方案： 

loop() ->
  receive
    {msg1, Msg1} -> 
      ...,
      loop();
    stop ->
      io:format("Server going down");
    Other ->
      error_logger:log({error, {process_got_other, self(), Other}}),
      loop()
  end. % 这是个尾递归

If you use some kind of server library, for example generic, you automatically avoid doing this mistake. 如果你使用了某种服务库，例如generic，你就可以自动避免这种错误。

5.10 接口函数

Use functions for interfaces whenever possible, avoid sending messages directly. Encapsulate message passing into interface functions. There are cases where you can't do this. 只要有可能，尽量使用函数作为接口，避免直接传递消息。封装消息到接口函数，有几种情况不能这么用。

The message protocol is internal information and should be hidden to other modules. 消息协议是内部信息，需要对外部模块隐藏。

Example of interface function: 接口函数示例： 

-module(fileserver).
-export([start/0, stop/0, open_file/1, ...]).

open_file(FileName) ->
  fileserver ! {open_file_request, FileName},
  receive
    {open_file_response, Result} -> Result
  end.

...<code>...

5.11 超时

Be careful when using after in receive statements. Make sure that you handle the case when the message arrives later (See "Flush unknown messages" on page 16.). 要注意，在receive代码段中使用after 。确定你处理消息返回延迟后的情况（参见"处理未知消息"）

5.12 捕获退出

As few processes as possible should trap exit signals. Processes should either trap exits or they should not. It is usually very bad practice for a process to "toggle" trapping exits. 尽可能少的进程可以捕获退出信号。进程要么捕获退出信号要么不能。一个进程"toggle"捕获退出信号通常是个糟糕的实践。

6 多种Elang特定的约定

6.1 使用记录为基础数据结构

Use records as the principle data structure. A record is a tagged tuple and was introduced in Erlang version 4.3 and thereafter (see EPK/NP 95:034). It is similar to struct in C or record in Pascal. 使用记录作为基础数据结构。记录是带标签的元组，从Erlang 4.3 版本之后开始支持。它类似C当中的struct或者Pascal中的record。

If the record is to be used in several modules, its definition should be placed in a header file (with suffix .hrl) that is included from the modules. If the record is only used from within one module, the definition of the record should be in the beginning of the file the module is defined in. 如果记录被多个模块使用，它的定义就应该放到头文件中（带.hrl后缀）然后被其他模块包含。如果记录只在一个模块中使用，那么定义应该放在文件开头，模块定义之内。

The record features of Erlang can be used to ensure cross module consistency of data structures and should therefore be used by interface functions when passing data structures between modules. Erlang的记录功能可以用来确保数据结构跨模块的一致性，因此应该在模块之间通过接口函数传递数据结构时使用。

6.2 使用选择器和构建器

Use selectors and constructors provided by the record feature for managing instances of records. Don't use matching that explicitly assumes that the record is a tuple. Example: 使用记录功能提供的选择器和构建器管理记录实例，不要假设记录是元组进行显式匹配。例如： 

demo() ->
  P = #person{name = "Joe", age = 29},
  #person{name = Name1} = P,% 使用匹配，或者...
  Name2 = P#person.name. % 像这样使用选择器

Don't program like this: 不要这样编程： 

demo() ->
  P = #person{name = "Joe", age = 29},
  {person, Name, _Age, _Phone, _Misc} = P. % 千万不要这样

6.3 使用带标签的返回值

Use tagged return values. 使用带标签的返回值。

Don't program like this: 不要这样编程： 

keysearch(Key, [{Key, Value}|_Tail]) ->
  Value; %% 不要返回不带标签的值！
keysearch(Key, [{_WrongKey, _WrongValue} | Tail]) ->
  keysearch(Key, Tail);
keysearch(Key, []) ->
  false.
Then the {Key, Value} cannot contain the false value. This is the correct solution: 这样的话{Key, Value}是不能包含false值的，这是正确的方案： 

keysearch(Key, [{Key, Value}|_Tail]) ->
  {value, Value}; %% 正确，返回了一个带标签的值
keysearch(Key, [{_WrongKey, _WrongValue} | Tail]) ->
  keysearch(Key, Tail);
keysearch(Key, []) ->
  false.

6.4 非常小心使用try...catch

Do not use catch and throw unless you know exactly what you are doing! Use catch and throw as little as possible. 不要使用catch和throw除非你清楚自己在做什么！尽可能少的使用catch和throw。

Catch and throw can be useful when the program handles complicated and unreliable input (from the outside world, not from your own reliable program) that may cause errors in many places deeply within the code. One example is a compiler. catch和throw在程序处理复杂且不可靠的输入（从外界，不是我们可信的程序）时非常有用。一个例子是编译器。

6.5 非常小心使用进程字典

Do not use get and put etc. unless you know exactly what you are doing! Use get and put etc. as little as possible. 不要使用get和put之类的除非你清楚自己在做什么！尽可能少的使用get和put之类的。

A function that uses the process dictionary can be rewritten by introducing a new argument. 一个需要使用进程字典的函数可以通过引入新参数的方式重写。

Example: 例如：
Don't program like this: 不要这样编码： 

tokenize([H|T]) ->
  ...;
tokenize([]) ->
  case get_characters_from_device(get(device)) of % 不要使用 get/1!
    eof -> [];
    {value, Chars} ->
      tokenize(Chars)
  end.

The correct solution: 正确的解决方案： 

tokenize(_Device, [H|T]) ->
  ...;
tokenize(Device, []) ->
  case get_characters_from_device(Device) of     % 这样更好
    eof -> [];
    {value, Chars} ->
      tokenize(Device, Chars)
  end.

The use of get and put will cause a function to behave differently when called with the same input at different occasions. This makes the code hard to read since it is non-deterministic. Debugging will be more complicated since a function using get and put is a function of not only of its argument, but also of the process dictionary. Many of the run time errors in Erlang (for example bad_match) include the arguments to a function, but never the process dictionary. 使用 get 和put会导致函数传入相同的参数时在不同场合有不同行为。这种不确定性会使代码可读性变差。在函数使用get 和put之后调试也会变复杂，当函数不再将参数作为唯一来源，还包含进程字典。Erlang运行时报错（例如bad_match）包含参数信息，但不包含进程字典。

6.6 别用import

Don't use -import, using it makes the code harder to read since you cannot directly see in what module a function is defined. Use exref (Cross Reference Tool) to find module dependencies. 不要使用-import，这样使用在你不能直接看到模块函数定义的时候降低代码可读性。使用exref来查看模块依赖性关系。（by kaiz: 应该是xref）

6.7 导出函数

Make a distinction of why a function is exported. A function can be exported for the following reasons (for example): 先看一下为什么要导出函数的差异。一个函数因为如下原因需要导出（举个例子）：

  • It is a user interface to the module. 它是模块的用户接口。

  • It is an interface function for other modules. 它是对其它模块的接口函数。

  • It is called from apply, spawn etc. but only from within its module. 它被apply, spawn之类的函数调用，但只在模块自身中。

Use different -export groupings and comment them accordingly. Example: 使用不同的-export将它们分组并增加对应注释，例如： 

%% 用户接口
-export([help/0, start/0, stop/0, info/1]).

%% 模块导出
-export([make_pid/1, make_pid/3]).
-export([process_abbrevs/0, print_info/5]).

%% 模块内调用
-export([init/1, info_log_impl/1]).

7 具体的词法和语法约定

7.1 不要写深度嵌套代码

Nested code is code containing case/if/receive statements within other case/if/receive statements. It is bad programming style to write deeply nested code - the code has a tendency to drift across the page to the right and soon becomes unreadable. Try to limit most of your code to a maximum of two levels of indentation. This can be achieved by dividing the code into shorter functions. 嵌套代码是一种包含case/if/receive代码段中还包含其他case/if/receive代码段的代码。编写深度嵌套代码是一种糟糕的编程风格，代码会越来越偏右，很快变得不可读。尝试大部分代码限制到最大缩进两个层次。这可以通过将代码分割成较短的函数来实现。 （by kaiz: 考虑到顶层业务逻辑比较复杂的情况，我个人建议还是将顶层逻辑放在一个函数中保持它的可读性，同时将正常逻辑（或者原理上最经常要执行的逻辑）放在最上，保证阅读时逻辑的通顺性，底层业务逻辑则可以视情况分割成若干函数）

7.2 不要写非常大的模块

A module should not contain more than 400 lines of source code. It is better to have several small modules than one large one. 一个模块代码不应该超过400行，把大模块分成若干小模块往往会更好。

7.3 不要写非常长的函数

Don't write functions with more than 15 to 20 lines of code. Split large function into several smaller ones. Don't solve the problem by writing long lines. 不要写超过15到20行代码的函数。将大函数分割为若干小函数。不要写很长的语句来解决问题。

7.4 不要写非常长的语句

Don't write very long lines. A line should not have more than 80 characters. (It will for example fit into an A4 page.) 不要写非常长的语句。一行代码应该不超过80字符（例如它将适合一张A4纸。）

In Erlang 4.3 and thereafter string constants will be automatically concatenated. Erlang 4.3版本以上支持字符串常量自动连接。 例如: 

io:format("Name: ~s, Age: ~w, Phone: ~w ~n" 
      "Dictionary: ~w.~n", [Name, Age, Phone, Dict])

7.5 变量命名

Choose meaningful variable names - this is very difficult. 选择一个含义丰富的变量名通常都很难。

If a variable name consists of several words, use "_" or a capitalized letter to separate them. Example: My_variable or MyVariable. 如果一个变量名由多个单词组成，需要用"_"或者首字母大写来分隔它们，例如：My_variable或者 MyVariable.

Avoid using "_" as don't care variable, use variables beginning with "_" instead. Example: _Name. If you at a later stage need the value of the variable, you just remove the leading underscore. You will not have problem finding what underscore to replace and the code will be easier to read. 避免使用"_"作为通配变量，使用以"_"开头的变量名称，例如：_Name。如果你稍候需要使用变量的值，你只需要删除开头的下划线。替换掉这些下划线不会有任何问题，也会让代码变得更易读。

7.6 函数命名

The function name must agree exactly with what the function does. It should return the kind of arguments implied by the function name. It should not surprise the reader. Use conventional names for conventional functions (start, stop, init, main_loop). 函数名称必须与函数所做的事情一致。函数的返回值应该在函数名中有暗示。函数名称不应该让读者惊异。传统功能则使用常用名字（start, stop, init, main_loop）。

Functions in different modules that solves the same problem should have the same name. Example: Module:module_info(). 在不同模块中实现同样功能的函数应该使用同样的名称。例如：Module:module_info()。

Bad function names is one of the most common programming errors - good choice of names is very difficult! 糟糕的函数名是一种最常见的编程错误 - 选个好的函数名的确不容易！

Some kind of naming convention is very useful when writing lots of different functions. For example, the name prefix "is_" could be used to signify that the function in question returns the atom true or false. 在编写大量不同的函数时，一种统一的命名约定会起到非常大的帮助。例如，以"is_"为前缀的命名可以用来标示处理这个问题的函数会返回一个原子是true或者false。 

is_...() -> true | false
check_...() -> {ok, ...} | {error, ...}

7.7 模块命名

Erlang has a flat module structure (i.e. there are not modules within modules). Often, however, we might like to simulate the effect of a hierarchical module structure. This can be done with sets of related modules having the same module prefix. Erlang的模块结构是平面化的（也就是说没有内嵌模块）。然而，通常我们可能希望模拟分层模块结构的效果。这可以通过让相关模块集拥有同样的前缀来实现

If, for example, an ISDN handler is implemented using five different and related modules. These module should be given names such as: 例如，假设一个ISDN处理器是由5个不同但又相关的模块实现的，这些模块应该这样命名： 

isdn_init 
isdn_partb 
isdn_...

7.8 风格一致的编程

A consistent programming style will help you, and other people, to understand your code. Different people have different styles concerning indentation, usage of spaces etc. 保持一致的编程风格会让你和其他人，更好的理解你的打码。不同的人有不同风格的缩进，空格用法等等。

For example you might like to write tuples with a single comma between the elements: 例如你在写元组的时候喜欢在元素之间只用一个逗号： 

{12,23,45}

Other people might use a comma followed by a blank: 其他人可能喜欢用逗号加空格： 

{12, 23, 45}

Once you have adopted style - stick to it. 一旦你适应了一个风格就坚持使用。

Within a larger project, the same style should be used in all parts. 在一个大项目中，应该使用同一种风格。

8 为代码提供文档

8.1 属性代码

You must always correctly attribute all code in the module header. Say where all ideas contributing to the module came from - if your code was derived from some other code say where you got this code from and who wrote it. 你必须经常为模块头正确标记属性。说得清有助于模块的所有想法是从哪里来的 - 如果你的代码是从一些其他的代码衍生说你来自哪里得到这个代码是谁写的。

Never steal code - stealing code is taking code from some other module editing it and forgetting to say who wrote the original. 不要窃取代码 - 指的是从其他模块拷贝过来代码却忘记标识谁是原来的作者。

Examples of useful attributes are: 

-revision('Revision: 1.14 ').
-created('Date: 1995/01/01 11:21:11 ').
-created_by('eklas@erlang').
-modified('Date: 1995/01/05 13:04:07 ').
-modified_by('mbj@erlang').
(by kaiz : 欢迎给自己的代码加上属性注释，暂不做硬性要求)

8.2 在代码中提供引用的规范

Provide cross references in the code to any documents relevant to the understanding of the code. For example, if the code implements some communication protocol or hardware interface give an exact reference with document and page number to the documents that were used to write the code. 提供有助于理解代码的代码交叉引用文档。例如，如果代码实现了某个通讯协议或硬件接口，最好在代码中标识具体引用的是哪个文档哪一页的内容。

8.3 为错误提供文档

All errors should be listed together with an English description of what they mean in a separate document (See "Error Messages" on page 32.) 所有的错误应该一起列在一份单独的文档中（参见"错误消息"）

By errors we mean errors which have been detected by the system. 这里的错误指的是系统检测到的错误。

At a point in your program where you detect a logical error call the error logger thus: 当你的程序检测到一个逻辑错误的时候可以这样调用错误记录器： 

error_logger:error_msg(Format, {Descriptor, Arg1, Arg2, ....})

And make sure that the line {Descriptor, Arg1,...} is added to the error message documents. 确保{Descriptor, Arg1,...}中的内容已经添加到错误消息文档中。

8.4 为消息中的所有基础数据结构提供文档

Use tagged tuples as the principle data structure when sending messages between different parts of the system. 在系统不同部分传递消息时应该使用带标签的元组作为基础数据结构。

The record features of Erlang (introduced in Erlang versions 4.3 and thereafter) can be used to ensure cross module consistency of data structures. Erlang的记录功能（自4.3版引入）可以用来保证跨模块数据结构的一致性。

An English description of all these data structure should be documented (See "Message Descriptions" on page 32.). 所有的数据结构应当有一份文档（参见"消息描述"）

8.5 注释

Comments should be clear and concise and avoid unnecessary wordiness. Make sure that comments are kept up to date with the code. Check that comments add to the understanding of the code. Comments should be written in English. 注释应该简洁明了，避免不必要的赘言。确保最新的代码都有注释。检查代码对应注释的含义。

Comments about the module shall be without indentation and start with three percent characters (%%%), (See "File Header, description" on page 29.). 模块注释应当没有缩进且以三个百分号（%%%）开始(参见"文件头描述")

Comments about a function shall be without indentation and start with two percent characters (%%), (See "Comment each function" on page 27.). 函数注释应当没有缩进且以两个百分号(%%)开始（参见"为每个函数提供注释" on page 27.）

Comments within Erlang code shall start with one percent character (%). If a line only contains a comment, it shall be indented as Erlang code. This kind of comment shall be placed above the statement it refers to. If the comment can be placed at the same line as the statement, this is preferred. Erlang代码中的注释应该以一个百分号(%)开始。如果一行只有一句注释，应该和Erlang代码一样缩进。这种注释应当在引用到的语句上方。注释最好能和注释的语句放在同一行。 

%% 函数注释
some_useful_functions(UsefulArgugument) ->
  another_functions(UsefulArgugument),    % 行尾注释
  % complicated_stmnt的注释，应当保持缩进
  complicated_stmnt,
......

8.6 为每个函数提供注释

The important things to document are: 需要编写文档的重要事项有：

  • The purpose of the function. 函数的目的。

  • The domain of valid inputs to the function. That is, data structures of the arguments to the functions together with their meaning. 函数的有效输入域。包括函数参数的数据结构和他们的含义。

  • The domain of the output of the function. That is, all possible data structures of the return value together with their meaning. 函数的输出域。包括所有可能返回的数据结构和它们的含义。

  • If the function implements a complicated algorithm, describe it. 如果函数实现了一个复杂算法，需要描述一下。

  • The possible causes of failure and exit signals which may be generated by exit/1, throw/1 or any non-obvious run time errors. Note the difference between failure and returning an error. 可能产生的错误或退出信号，由exit/1, throw/1或者其他不明显的运行时错误产生的消息。注意区分函数执行失败和返回错误。

  • Any side effect of the function. 函数的副作用。

例如： 

%%----------------------------------------------------------------------
%% Function: get_server_statistics/2
%% Purpose: Get various information from a process.
%% Args:   Option is normal|all.
%% Returns: A list of {Key, Value} 
%%     or {error, Reason} (if the process is dead)
%%----------------------------------------------------------------------
get_server_statistics(Option, Pid) when pid(Pid) ->
  ......

8.7 数据结构

The record should be defined together with a plan text description. Example: 

%% File: my_data_structures.h

%%---------------------------------------------------------------------
%% Data Type: person
%% where:
%%    name: A string (default is undefined).
%%    age: An integer (default is undefined).
%%    phone: A list of integers (default is []).
%%    dict:     A dictionary containing various information about the person. 
%%       A {Key, Value} list (default is the empty list).
%%----------------------------------------------------------------------
-record(person, {name, age, phone = [], dict = []}).

If you know of any weakness, bugs, badly tested features, make a note of them in a special comment, don't try to hide them. If any part of the module is incomplete, add a special comment. 9 常犯错误：

  • Writing functions which span many pages (See "Don't write very long functions" on page 23.). 编写跨了很多页的函数(参见"不要写非常长的函数")

  • Writing functions with deeply nested if's receive's, case's etc (See "Don't write deeply nested code" on page 23.). 编写深度嵌套if、receive、case等的函数。(参见"不要深度嵌套代码")

  • Writing badly typed functions (See "Use tagged return values" on page 19.). 编写糟糕返回类型的函数(参见"使用带标签的返回值")

  • Function names which do not reflect what the functions do (See "Function names" on page 24.). 函数名称并未反应其作用(参见"函数命名" on page 24.)

  • Variable names which are meaningless (See "Variable names" on page 23.). 变量名称毫无意义(参见"变量命名")

  • Using processes when they are not needed (See "Assign exactly one parallel process to each true concurrent activity in the system" on page 14.). 使用多余的进程(参见"为系统中每一个活动分配唯一的进程")

  • Badly chosen data structures (Bad representations). 糟糕的数据类型选择（糟糕的数据表现）。

  • Bad comments or no comments at all (always document arguments and return value). 糟糕的注释或者完全没有注释（参数和返回值总要有吧）。

  • Unindented code. 未缩进的代码。

  • Using put/get (See "Use the process dictionary with extreme care" on page 20.). 使用了 put/get (参见"非常小心使用进程字典")

  • No control of the message queues (See "Flush unknown messages" on page 16. and See "Time-outs" on page 18.). 没有对消息队列进行控制(参见"处理未知消息" 和 参见 "超时")
